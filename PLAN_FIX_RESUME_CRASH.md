# PLAN: Fix Systemic Resumption Crash (RemoveMessage ID Conflict)

## 1. Problem Statement
When the TradingAgents graph is resumed from a seeded checkpoint (e.g., `analysts_checkpoint.json`), the `messages` list may contain `RemoveMessage` objects generated by "Msg Clear" nodes in the previous execution. 
- LangGraph's `MessagesState` reducer attempts to apply these `RemoveMessage` objects to the current history.
- Because the resume starts a fresh graph instance with an empty volatile history, the target message IDs do not exist.
- This causes a fatal crash: `Attempting to delete a message with an ID that doesn't exist`.

## 2. Research Findings
- **Trigger**: The crash consistently happens during Phase 2/3 resumption (e.g., at `Bear Researcher`).
- **Data**: `analysts_checkpoint.json` for `SHOP` confirmed the presence of `RemoveMessage` objects in the `messages` array.
- **Scope**: This is a systemic issue affecting any ticker that reaches the "Msg Clear" nodes and is subsequently resumed.

## 3. Proposed Fix
Modify `agent_os/backend/services/run_helpers.py:repair_checkpoint_messages` to filter out ALL `RemoveMessage` objects.
- **Justification**: For a seeded resume, the `messages` list already represents the state *after* pruning. The `RemoveMessage` objects are "tombstones" that are only valid for the specific execution thread where they were created.
- **Safety**: Removing them is safe because the messages they were intended to delete are already absent from the list. It prevents the engine from attempting "double-deletions" on non-existent IDs in the new instance.
